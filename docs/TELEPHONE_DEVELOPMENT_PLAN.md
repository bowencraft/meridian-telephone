# Telephone 完整开发计划

最后更新：2026-07-17  
项目目录：当前仓库根目录
参考项目：同级 Typewriter 项目

## 目标

Telephone 是 Typewriter“单物品叙事工具链”的第二个完整游戏样本。项目要保留 Typewriter 的开发流程、后台图编辑器、规则驱动剧情系统与本地运行方式，同时把核心交互从“打字机输入”完整重构为“老式电话接拨与通话分支”。

最终交付不能停留在可玩原型：它必须是一款极致精美、内容充足、可反复体验的网页叙事解谜游戏，拥有完整前台、可视化剧情后台、多周目差异、多结局、完善音画反馈、响应式适配、自动化测试与构建验证。

## 项目定位

Telephone 是一个发生在 20 世纪伦敦雨夜公共电话亭中的 2D 拟物解谜游戏。玩家面对一部老式转盘电话，通过接听来电、拨打号码、选择回应、监听文字化通话内容，逐渐发现一家神秘推销公司正在利用电话网络洗脑接听者，把他们转化为新的推销员。

它与 Typewriter 一样，是“单物品强交互 + 节点图叙事 + 后台可编辑”的网页解谜游戏：

- Typewriter 的核心是纸张长度和自由文本审讯。
- Telephone 的核心是电话状态机、号码拨号、来电事件、通话选项、超时挂断与碎片化电话网络剧情。

## 故事包装

游戏发生在伦敦某个雨夜，年代约为 1960s 至 1980s 之间一个有意模糊的二十世纪。玩家进入街边公共电话亭。亭外有雨水、车灯、远处人声与霓虹反光；亭内贴着小广告、便签、手写号码、失物招领、过期演出海报、报纸剪角与奇怪的推销传单。

表层体验是接打电话：有人询问天气，有人声称自己迷路，有人留下毫无上下文的号码，有人像自动语音一样重复推销话术。随着进展，玩家会发现许多来电都与同一家公司有关。

### 核心公司

暂定名：**The Meridian Courtesy Exchange（子午礼仪交换所）**

- 表面业务：电话礼仪培训、天气信息服务、家庭推销、客户满意度调查。
- 真实目的：通过重复话术、等待音、转接流程、号码选择与“礼貌回应训练”，慢慢重写接听者的语言习惯和身份认知。
- 最终结果：受影响的人会主动拨打陌生号码，成为新的推销员。

### 四幕主线

#### 第一幕：普通电话亭

玩家学会提起听筒、拨号与接听来电。电话内容看似日常：天气、路线、找人、错误号码。

#### 第二幕：推销话术入侵

一些来电开始要求玩家重复固定句式，例如“我愿意听取更好的选择”。拨错号码时偶尔不再是空号，而会进入奇怪的自动菜单。

#### 第三幕：电话网络显形

墙上小广告、号码纸条与来电内容开始互相指向。玩家发现不同陌生人说出相同句子，像被同一套脚本替换。

#### 第四幕：反向拨号 / 逃离或接管

玩家利用收集到的号码和分支信息打入公司内部系统。最终可以切断交换机、被招募、假装加入后反制，或成为新的电话亭声音。

### 核心结局

第一版至少包含以下五个结局，完整产品应通过隐藏信息、顺序差异与跨周目记忆提高重玩价值：

1. **成功结局：断线**  
   玩家找到核心总机号码并完成正确分支，使公司电话网络短暂停摆。

2. **洗脑结局：入职**  
   玩家多次选择顺从、礼貌并重复推销句式，最终屏幕显示“欢迎加入 Meridian”。

3. **迷失结局：永远转接**  
   玩家陷入自动菜单循环，通话时长不断增长，无法回到现实。

4. **空号结局：号码磨损**  
   玩家频繁拨错、超时与挂断，电话亭逐渐只回应空号提示。

5. **模糊结局：天气很好**  
   玩家已知公司有问题却没有找到总机，只接到最后一通关于天气的电话，暗示自己已受影响。

## 核心交互计划

### 电话状态机

前台体验围绕清晰、可测试的状态机实现：

- `idle`：听筒挂着，电话可主动响铃。
- `ringing`：电话响铃，等待玩家提起听筒。
- `offHook`：玩家提起听筒，可以拨号或进入已接通来电。
- `dialing`：玩家拖动转盘输入号码。
- `connecting`：拨号后等待接通，可能出现拨号音、忙音、空号或剧情接入。
- `inCall`：通话中，显示对方文字，玩家可选择回应。
- `awaitingChoice`：云朵选项浮现在听筒右侧。
- `timeoutWarning`：长时间无操作，以画面或声音警告。
- `hungUp`：电话自动或手动挂断，清理当前通话。
- `ending`：进入结局记录页或结局状态。

### 转盘拨号

转盘必须单独建模，不能退化为数字按钮：

- 每个数字拥有对应孔位。
- 玩家按住某个数字孔并顺时针拖动至挡片位置。
- 松手后转盘机械回弹。
- 回弹完成才确认该数字。
- 数字 `0` 的旋转距离最长。
- 拖动方向错误、中途松手或拖动不足时不录入数字，并播放轻微机械失败音。
- 每次成功输入后，LCD 实时显示当前号码。
- 同时支持鼠标、触摸与键盘无障碍替代输入，但视觉与声音反馈仍保持转盘语义。

### 听筒

- 点击或拖动听筒均可提起。
- 提起后听筒随指针小范围悬浮。
- 移动范围受电话线长度限制。
- 松开后若靠近挂钩则挂断，否则保持悬浮或手持状态。
- 通话中挂断会中断当前分支。
- 电话响铃时提起听筒进入来电节点。
- 提起、落回、强制挂断均具有不同力度的声画反馈。

### LCD 屏幕

- 待机显示时间、雨夜干扰或 `INSERT COIN` 风格文字。
- 拨号时显示输入号码。
- 接通时显示通话时长。
- 空号时显示 `NO LINE` / `NUMBER UNOBTAINABLE`。
- 重要剧情时短暂显示异常文本，如 `YOU ARE BEING CONNECTED`、`SMILE WHILE SPEAKING`。
- 使用扫描线、余辉、轻微坏点、亮度波动与字符切换动效表现老式改装数码屏。

### 云朵选项

- 剧情需要玩家回复时，在听筒右侧浮现 2 至 4 个云朵选项。
- 每个选项对应一条图边。
- 选项可具有隐藏状态条件，例如仅在获得某个号码后出现。
- 选择后进入下一节点，并播放低语、线路噪声或轻微电流音。
- 超时未选择可触发沉默、挂断或隐藏分支。

### 场景交互

- 电话亭墙面、小广告、号码纸条、硬币槽、电话簿、玻璃窗雨痕均为独立交互对象。
- 这些对象可设置 flags、揭示号码、改变可拨号列表、触发支线来电或提供隐藏线索。
- 场景交互逻辑集中在 `sceneInteractions` 模块，不与电话拨号状态机耦合。
- 热点要有细腻但克制的悬停、聚焦、已检查与发现状态。

## 剧情数据结构

Telephone 继续采用“唯一 JSON 剧情源文件”的原则，并直接靠近通用 `GraphDocument` 格式，以便未来 Typewriter 与 Telephone 共用编辑器。

```ts
interface TelephoneStory {
  format: 'graph-content'
  formatVersion: number
  id: string
  title: string
  entryNodeId: string
  nodes: TelephoneNode[]
  edges: TelephoneEdge[]
  globals: {
    timeout: {
      dialIdleMs: number
      choiceIdleMs: number
      callMaxMs?: number
    }
    phone: {
      validNumbers: string[]
      emergencyNumbers?: string[]
      wrongNumberNodeId: string
      idleRingSchedule?: RingEvent[]
    }
  }
  extensions: {
    telephone: {
      lcdMessages: Record<string, string>
      endings: Record<string, string>
      sceneHotspots: SceneHotspot[]
      audioProfile: AudioProfile
    }
  }
}
```

### 节点类型

- `start`：初始状态
- `idle`：电话亭待机剧情节点
- `incoming_call`：主动来电
- `dial_target`：拨号可达节点
- `call`：普通通话节点
- `menu`：自动语音菜单
- `scene`：场景发现节点
- `global`：任意状态可触发
- `end`：成功或特殊结局
- `fail`：失败结局

### 边触发条件

- `dialNumber`：拨出指定号码
- `choice`：点击指定云朵选项
- `incomingAnswer`：响铃时接起
- `hangUp`：玩家挂断
- `timeout`：超时
- `sceneInspect`：点击场景物件
- `stateEquals`：检查 flags
- `keywordAny`：保留给后台搜索与文本式测试

后台应能同时编辑：

- 哪些号码有效
- 错号进入哪个节点
- 来电何时出现
- 每个通话节点的对方文本
- 玩家可选回复
- 选择后的 flags
- 超时跳转
- 场景物件触发的状态变化

## 美术与 2D 场景计划

第一版保持 2D，不使用 Three.js。

### 视觉目标

- 公共电话亭内部作为固定主场景。
- 电话为中心大物件，占画面主要区域。
- 背景包含玻璃雨痕、暗红电话亭框架、昏黄灯光与墙上广告。
- LCD、拨号盘、听筒与电话线具有清晰交互状态。
- 不堆叠说明文字；交互反馈主要依靠光效、震动、声音与鼠标状态。
- 画面应具有真实材质层次、戏剧性照明与统一时代感，同时保证文本和交互可读性。

### 分层绘制

- 背景层：电话亭墙、玻璃、雨、广告。
- 电话机层：机身、转盘、硬币槽、LCD。
- 交互层：听筒、电话线、云朵选项、热点高亮。
- 氛围层：雨水反光、灯闪、线路干扰、LCD 扫描感。

## 开发阶段

### 1. 项目初始化

从 Typewriter 结构复用 Vite / React / TypeScript 配置，建立 Telephone 文件命名、路由、样式入口与测试脚手架。

### 2. 核心电话状态机

实现听筒、挂断、拨号、接通、空号、超时与来电状态，先以可测试逻辑闭合完整流程。

### 3. 转盘拨号模型

实现真实拖拽、角度计算、回弹确认、数字输入与错误拖动处理，并为 `dialModel` 编写单元测试。

### 4. 剧情引擎适配

把 Typewriter 的文本匹配图引擎改造为事件驱动图引擎，支持 `dialNumber`、`choice`、`timeout`、`hangUp`、`incomingAnswer` 与 `sceneInspect`。

### 5. 前台完整体验

完成电话亭 2D 场景、LCD、通话文本、云朵选项、音效、电话线约束与主动响铃。

### 6. 后台适配

复用 Twine 图编辑器思路，替换 Inspector 为 Telephone 字段，加入号码簿、超时、来电与场景热点配置。

### 7. 完整示例剧本

不止实现最小 12 至 18 节点示例，而要完成内容充足、分支交织、可重复体验的推销电话主线，至少含 5 个可拨号码、3 个主动来电、5 个核心结局、支线、隐藏发现与跨周目变化。

### 8. 测试与精修

覆盖状态机、拨号模型、图匹配、超时、记录生成、持久化与剧情可达性；通过浏览器实测拖拽、触摸、常见视口、音效释放、后台保存与全流程结局。

## 第一版剧本骨架

开局电话亭待机，墙上可发现三个号码：

- 天气服务号码
- Meridian 免费咨询号码
- 被划掉的内部转接号码

三通核心主动来电：

1. 陌生人询问：“今天伦敦是不是在下雨？”
2. 自动语音播报：“感谢您选择 Meridian Courtesy Exchange。”
3. 一名疲惫的推销员提醒玩家不要重复它要求玩家重复的话。

玩家逐步获得内部号码，通过拨号进入自动菜单。菜单要求玩家选择：

- 听取产品介绍
- 申请成为代表
- 投诉线路问题
- 查询天气

正确路线是从“投诉线路问题”进入人工转接，再利用场景中发现的旧编号进入总机。最终揭示所有天气服务、错误号码和推销电话都连接到同一个语音训练系统。

## 可反复体验要求

- 各次游戏中主动来电的时间与部分文案可在规则约束下变化。
- 不同检查顺序会提前或延后解锁号码和回应。
- 玩家顺从、怀疑、粗暴、沉默与好奇等倾向应被 flags 记录，并改变后续话术。
- 结局后保留已发现号码、结局档案或异常残留，形成轻度跨周目记忆。
- 至少一个隐藏真结局需要综合多周目信息，但不能以机械重复刷取为前提。
- 通话文本应有足够 variants，避免同一路径每次完全相同。
- 记录页要鼓励再次进入电话亭寻找未发现号码与结局。

## 完整验收标准

### 前台

- 玩家可提起和拖动听筒，并受电话线长度限制。
- 玩家可真实拖动转盘拨号并看到机械回弹。
- 正确号码能够接通；错误号码进入空号、忙音或剧情性错号流程。
- 电话会主动响铃，玩家可接听、拒接或错过。
- 通话文字清晰显示，玩家通过云朵选项回应。
- LCD 实时显示号码、状态、异常消息与通话时长。
- 长时间无操作会警告并自动挂断或进入剧情超时。
- 场景热点可点击，并能设置 flags、发现号码或触发支线。
- 支持桌面、触摸设备和窄屏视口；键盘操作具备可用的替代路径。
- 声音可单独启停，首次交互后正确释放音频上下文。

### 剧情与重玩

- 故事有完整四幕推进，而非单次线性电话。
- 至少 5 个有效可拨号码、3 个主动来电和 5 个核心结局。
- 所有核心结局可达，主要分支没有死路或不可退出循环。
- 有足量通话 variants、隐藏选择、场景发现、支线和多周目变化。
- 有一个完整成功结局以及多个失败、模糊或隐藏结局。

### 后台

- 可编辑节点、转场、号码、选项、超时、来电排程与场景热点。
- 可在图上创建、移动、连接和删除内容，并保留坐标。
- 可模拟拨号、来电、选项、场景检查、超时和挂断。
- 可查看当前 node、flags、候选边、通话记录和 LCD 状态。
- 剧情 JSON 可导入导出。
- 本地开发可保存到源码，静态环境回退到 `localStorage`。
- `src/story/telephone.rules.json` 是唯一剧情源文件。

### 记录与持久化

- 结局后可打开通话记录页。
- 记录包含会话编号、日期、结局、拨号记录、通话摘要、重要选择与已发现号码。
- 结局档案与已发现内容可持久化，并支持清除进度重新开始。

### 工程质量

- 逻辑模块具备严格 TypeScript 类型。
- 状态机、转盘模型、事件匹配、超时、持久化、记录和剧情可达性具有自动化测试。
- `npm test`、`npm run lint` 与 `npm run build` 全部通过。
- 通过浏览器完成桌面与窄屏关键流程验收。
- Git 使用独立 `codex/` 功能分支，提交信息遵循常见约定式风格并按里程碑组织。

## 产品原则

Telephone 不是 Typewriter 的换皮，而是 Typewriter“单物品叙事工具链”的第二个完整游戏验证样本。开发它的同时，应推动图编辑器从打字机专用后台演进为更通用的交互叙事后台。

最终完成的页面应当首先是一款令人愿意反复进入电话亭、重新拨号和追查秘密的精致游戏，其次才是一套技术演示。
