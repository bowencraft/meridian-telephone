# Seedline游戏数据执行大纲

## Purpose

把六章正文转成玩家实际能操作的电话图。正文仍是情节与人物规范；`src/story/telephone.rules.json`是运行时唯一内容源。本大纲规定迁移顺序、底层补强、节点命名、章节出口和验收见证，避免只把小说段落塞进随机variants。

## Runtime Decisions

1. 一次完整游玩代表Vale依靠交班磁带连续恢复并处理六个夜班档案。章节日期由通话文本和LCD明确；页面不伪装真实经过六天。
2. 每章结束不弹“下一章”按钮。人物或工单清楚交付一个号码，玩家挂断、重新投币并拨该号码，亲手开启下一章。
3. 结局后的“再次进入”代表更晚的一次MCE-0复核夜班，不是时间回溯。第一次处理4.0首发；之后依次处理4.0-R1、R2等事故复核、地方接入或签名重开docket。上一结局已经造成的现实后果不撤销，但新的docket仍可作另一项治理决定，因此同一档案可以归档多个结局。
4. 章节内临时窗口、当前排队位置、当夜监听状态与未提交草稿使用runtime flags；`facts`只保存“历史上发生过”；`durableState`覆盖保存当前签名、来源卡、令牌、医院证据与发布状态；`lastEnding`保存上一夜结果。
5. 主线号码始终有固定口述或固定工单来源。概率物件只能提供重复来源、捷径、人物侧面或离散支线。

## Infrastructure Gate

在重写规则JSON前先补齐以下能力：

| 能力 | 数据/API | 玩家层理由 | 验收 |
| --- | --- | --- | --- |
| 跨周目事实 | `ProgressData.facts`、`RuntimeState.facts`、`hasFact`、`addFact` | 人物能记得帮助、背叛、损坏与版本后果 | 完成结局后重开，条件分支能读取既有事实 |
| 当前长期状态 | `durableState`、`durableEquals`、`setDurable`、`lastEndingEquals` | 累积历史之外还能回答“现在是什么状态”与“上一夜是什么结果” | 损坏→重建、令牌发放→消耗、批准→后续撤销均按最新状态分支 |
| 周目条件 | `attemptsGte` | 新闻、广播和交班磁带能随已完成夜班演化 | 第2次及以后进入出现不同内容 |
| 号码alias | `PhoneDirectoryEntry.aliases`与统一号码规范化 | 旧印刷号、内部简写和完整本地号指向同一联系人 | alias接入同一节点、号码簿不重复联系人 |
| 可持续投币 | 已接通或结束的呼叫把MCE-0审计币退回 | 六章和支线不会因1–3枚随机硬币软锁 | 任意合法呼叫后仍能继续主线；错号也有恢复 |
| 三层物件文案 | `copy.summary`、`copy.style`、`firstVariants/repeatVariants` | 无号码物件可分别说明它是什么、什么样、意味着什么 | clue card明确显示粗略、样式、细节 |
| 通话中物件锁 | 柜台物件与墙面物件同样只在idle可检查 | 防止UI处于通话但引擎被重置到idle | 组件/交互测试覆盖 |
| 完整验证 | ring、alias、号码来源、sceneEvent、可达性 | 后台结构通过不等于玩家可玩 | validator与路径测试同时通过 |
| 漏接恢复 | `RingEvent.missedEffects` | Maeve漏接后仍能登记医院号码与事件线索 | 13.5秒漏接后号码簿出现7350194 |
| 阶段新闻 | `SceneCandidate.priority` + `lastEndingEquals` | 只从最高已满足阶段抽取，不随机回到旧报 | 同一槽从基础报逐夜升级到上一结局报道 |
| Choice分组 | 相同`trigger.value`只显示最高优先级胜出边 | OR与“可用/缺条件”不生成重复按钮 | getChoices与validator测试 |

## Chapter Node Packages

### Chapter 1 — 雨夜转运

入口是Maeve固定首响来电 `ring_maeve_transfer`。漏接时该RingEvent的`missedEffects`固定登记医院总机7350194和事件线索；999与医院便笺是重复入口，便笺不承担唯一补救。节点包：

```text
ch1_maeve_alert
→ ch1_patient_status
→ ch1_route_task
→ dial 9460264
→ ch1_road_service
→ ch1_driver_original / ch1_central_classification
→ discover 8714000
```

玩家第一次必须知道：Thomas需要透析转运、A路口积水、中央称“无服务影响”、Vale要选择保留哪种记录。接受中央分类不会立即结束全游戏，但写入 `hospital-classified`；恢复司机原声写入 `hospital-original-restored` 与 `maeve-trust`。两路都产生承包方号码，保证剧情继续。

### Chapter 2 — 夜班员工

`8714000`由Peter在公共合同值班台接入。三个选择分别保留跨系统故障、改成重复查询、归责司机；每个选择都明确影响Peter纪律风险、医院证据质量和Maeve关系。

章节稳定出口是正式投诉 `8714003`。人物支线 `8714227`让玩家理解Peter的母亲照护津贴、团队奖金和分区合同；若玩家愿意共同署名，写入 `peter-allied` 和 `employee-testimony`。Peter不会因一次善意就交出最终令牌。

### Chapter 3 — 十九号席位

投诉台要求真实事件号、终端与故障现象。Leonard通过四秒人工窗接入，只确认工作声纹和技术记录，不授予创始人治理权。

玩家随后拨 `8714019`听三音检索与Vale口述交班。把身份当权限只得到拒绝与解释；按工单边界返回Leonard才获得 `work-self-confirmed`、`technical-log-valid`，以及两项外部来源号码 `3011968`、`7941966`。

### Chapter 4 — 外部托管

Dorothy与Radio Nocturne可以任意顺序拨打。前者核验托管条款，后者核验公开时间戳；每一方都不知道完整答案。

```text
3011968 → escrow-source-valid
7941966 → public-timestamp-valid
both → 19c-release-valid → discover 8714036
```

若上一周目存在 `source-card-damaged`，两条线路先分别提供重建步骤；完成两项仍可恢复，不造成永久软锁。

### Chapter 5 — 创始人原声

`8714036`拆成四段可选择播放的历史录音：1981公共成功、1984私有化、1986长期分区试验、1989六周拖延。玩家至少听完四类，或在后续周目凭durable facts跳过已归档段落，才进入Wren回应。

本章交付：双签名规则、地方副本边界、Vale责任与凌晨四点发布号 `8714127`。Wren的中央方案必须列出真实收益：兼容表、福利、减少错误转接；代价是事故降级和治理垄断。`callMaxMs`提高到至少300000、`choiceIdleMs`提高到至少60000，四段录音也允许分次重拨，不能被旧95秒总上限截断。

### Chapter 6 — 四点发布

发布台只组合前五章已见能力。入口先显示当前缺失项；没有证据时给出负责人和恢复号码，而不是忙音。

发布台每一轮先播放上一docket后果，再开启本轮新docket。七个互斥choice进入七个固定EndingType：

| 游戏EndingType | 正文结局 | 必要条件 | 核心动作 |
| --- | --- | --- | --- |
| `weather` | 天气很好 | `releaseGate=open` | 批准本轮中央兼容表并接受事故降级 |
| `recruited` | 入职 | `releaseGate=open` | 以本轮签名交换六周公司整合 |
| `transfer` | 永远转接 | `releaseGate=open` | 请求继续审查直至本轮04:00 |
| `worn` | 号码磨损 | `releaseGate=open` | 物理破坏来源卡与索引，写`sourceCard=damaged` |
| `counterfeit` | 假代表 | `releaseGate=open`且`peterToken=available` | 延迟六小时并写`peterToken=spent` |
| `disconnected` | 断线 | `releaseGate=open`且`safetySignature=released` | 撤销签名，写`safetySignature=revoked` |
| `operator` | 夜班接线员 | `releaseGate=open`、`hospitalEvidence=strong`、`employeeEvidence=valid`、`technicalEvidence=valid`、`custodyEvidence=valid` | 附条件发布并建立多方确认 |

`releaseGate=open`的唯一公式是：`workSelf=confirmed`、`founderHistory=complete`、`dualSignature=known`、`technicalEvidence=valid`、`custodyEvidence=valid`，且`sourceCard`不是`damaged`。第一次双来源核验完成节点 `ch4_custody_release` 同时写 `sourceCard=released` 与 `safetySignature=released`；损坏后双来源重建写`sourceCard=rebuilt`并重新写`safetySignature=released`。所有公式使用可覆盖durableState，不靠累积facts猜测当前状态。

## Early-Dial Contract

玩家可以猜中任何号码。每个登记号码都有三档显式边：

1. 高优先级：条件完整，进入当前章节内容。
2. 中优先级：已完成该章，返回随人物与上一结局变化的回访内容；发布台例外——回访摘要后始终可开启下一复核docket。
3. 低优先级：条件不足，说明缺哪份工单、来源或签名，并给出可行动的恢复号码。

不得让“已登记但条件不足”落入通用busy节点，也不得增加wrongDials。

## Text Migration Rules

- `body.variants`是同一信息的措辞变化，不是按顺序播放的段落。
- 每个节点只陈述一个行动问题；一段长通话拆成多个有选择的节点。
- 正常variants至少两条，语义与事实完全一致；fallbackVariants说明当前线路仍在等什么。
- 不用hidden choice承载玩家必须点到的内容。
- 相同choice value的条件分支由前台只显示最高优先级胜出边；低优先级边可解释条件缺口而不产生重复按钮。
- 无选项节点必须有可玩的choice、明确timeout边或允许挂断；不依赖`auto`。
- 所有技术说明后立刻改变一个选项、号码或证据状态。

## Verification Order

1. 类型、迁移、持久化、投币和通话中物件锁的单元测试。
2. 六章最短主路径测试。
3. 七个结局逐一由真实事件序列抵达的测试。
4. 使用同一ProgressData、不清档连续抵达至少两个、最终可依次归档七个结局的发布周期测试。
5. 早拨、漏接、挂断、错号、物件缺席和跨周目恢复测试。
6. validator、lint、build和JSON图统计。
7. 清理浏览器story override和进度，由不知道设定的Sub-agent盲玩多个结局。

## Exit Condition

每个结局都有一条从空档案开始、只用前台可执行动作生成的见证路径；任意主线号码遗漏、挂断、错拨或概率物件缺席都不会造成无法继续的idle状态；盲玩者能复述当前任务、下一号码来源与结局代价。
