# Quartz礼仪手册网站完整实施计划

## 1. 目标与建议命名

在不改变当前游戏运行数据的前提下，把 `story-outline/` 建成一个和 Telephone 共用世界与视觉语言、但可独立阅读的 Quartz 网站。网站同时承担三种阅读方式：

1. 互相连接的人物、号码、物件、线索与结局 Wiki；
2. 只呈现六章 `Draft` 正文的网页小说阅读区；
3. 按目录保留大纲、人物卡、设定、素材、实施记录与修订记录的幕后档案区。

建议公开路径：

```text
/handbook/
```

建议页面署名：

```text
MERIDIAN COURTESY EXCHANGE
NIGHT STAFF HANDBOOK · COPY 19
子午礼仪交换所 · 夜班职员手册 · 十九号副本
```

“Courtesy”表面是公司的服务承诺，实际也是它把服从包装成礼貌的制度语言；“Handbook”表面像员工手册，阅读深入后会变成十九号席位留下的反向档案。它比纯技术名 `/mce-19/` 更容易被读者理解，同时仍有隐藏含义。

可保留两个备选，不在第一版同时启用：

- `/night-docket/`：更像夜班案卷，调查感更强；
- `/mce-19/`：最隐蔽，但初次访问者不容易理解用途。

## 3. 正史整理原则

### 3.1 三层内容身份

为避免概念继续变复杂，网站只在幕后页面显示三种简短标记：

| 标记 | 含义 | 可用于哪些页面 |
| --- | --- | --- |
| 已部署 | 玩家在当前游戏 JSON 中能直接遇到的事实 | 电话、物件、来电、节点、结局 |
| 扩写 | 与已部署内容不冲突、用于补全人物动机和世界背景的叙事；网页显示时必须标注 `[网页未实装]` | 人物档案、章节正文、幕后设定 |
| 开发记录 | 已被恢复版本覆盖、仍有创作参考价值的旧方案；网页显示时必须标注 `[网页未实装]` | 实施记录、废案、旧号码方案 |

公开 Wiki 默认只显示“已部署”和不矛盾的“扩写”。所有扩写或开发记录在标题、摘要或事实卡上使用完全一致的 `[网页未实装]` 文案；“开发记录”只进入幕后档案，不参与默认关系图和电话簿。

### 3.2 决策优先级

发生冲突时按以下顺序处理：

1. 当前 `src/story/telephone.rules.json` 和当前固定柜台物件；
2. 当前活动游戏的玩家可见文案；
3. 能解释活动内容、但没有新增玩家事实的扩写设定；
4. 历史实施计划与未部署素材。

不反过来修改当前 JSON 去迎合扩写稿。本项目的第一步是让网站与恢复后的远端故事一致。

### 3.3 六章号码骨架的目标状态

扩写正文与 Wiki 最终共同使用下面这条容易理解的活动线路：

```text
946 0264 天气服务
  → 871 4000 Meridian公开咨询
  → 871 4003 投诉 / 871 4019 十九号席位
  → 301 1968 失物招领 + 794 1966 Radio Nocturne
  → 871 4127 中央交换机
  → 七结局
```

处理扩写号码时采用以下方式：

- Maeve若保留，使用主动来电或医院外部背景，不再依赖一条游戏中不存在的医院直线；
- Peter若保留，通过 `871 4000` 的代表/员工分支出现，不再拥有未部署的 `871 4227`；
- 创始人原声若保留，改由 `871 4019`、`794 1966` 或外部档案内容承载，不再新增 `871 4036`；
- 三条地方支线号码保留为幕后研究素材，不进入当前可拨电话簿；
- 七个结局的标题、描述与触发语义以当前 JSON 为准，扩写正文不能把同名结局改写成另一种结果。

### 3.4 人物与活动角色的对照方式

目前只有 Evelyn Vale 与“E. Vale / 十九号席位”存在强对应。其他人物与活动角色的对应关系需要逐条核验，不能仅因职业相似就直接宣布同一人。

实施时建立一份只供本地校对的人物对照表，每项记录：

- 活动游戏角色；
- 扩写人物；
- 共同证据；
- 冲突证据；
- 结论：同一人、可能同一人、独立人物或仅开发原型。

初始待核对假设包括：

- Peter 与疲惫推销员 / Meridian代表线；
- Leonard 与旧接线员 / 故障人工线；
- Dorothy 与失物管理员；
- Wren 与中央验证员 / 主管声音；
- Maeve 与陌生雨况来电者。

任何存在性别、职业、知识范围或电话号码冲突的对应都保持“可能”，或者拆成独立人物。该对照表放在 `handbook-site/internal/`，明确从 Quartz 同步与导航中排除；公开 Wiki 不提供“扩写人物对应谁”的专门页面，也不制造虚假确定性。

## 4. 信息架构

网站使用一个 Quartz 实例，但在阅读体验上分成两大部分，另给正文提供专门入口。

### 4.1 第一部分：十九号关系 Wiki

建议路径：

```text
/handbook/wiki/
```

一级栏目：

1. **人物与声音**：人物档案、机构角色、帮助/阻拦/路人身份、关系；
2. **线路与电话簿**：8 个活动号码、3 个主动来电、号码发现来源、可拨状态；
3. **物件与证据**：10 件活动物件，以及它们指向的人、线路、线索和结局；
4. **通话与剧情流转**：主线路、旁支、失败恢复、章间号码交接；
5. **七个结局**：前置选择、最终状态、重复游玩的遗留影响；
6. **故事正文**：六章正文入口、上一章/下一章、章节涉及的号码与人物；
7. **词汇与时间线**：只解释读者必须知道的术语，不把所有幕后概念堆在首页。

每个 Wiki 页面统一先回答四件事：

1. 一句话说明它是什么；
2. 玩家在当前游戏里实际看见什么；
3. 扩写稿补充了什么；
4. 它链接到哪些人物、物件、号码、章节或结局。

### 4.2 网页小说阅读区

建议路径：

```text
/handbook/story/
```

这里不是第三套手写内容。构建脚本从 `chapters/*.md` 的唯一 `## Draft` 区间生成六个阅读页：

- 不显示 Synopsis 与 Revision Notes；
- 保持章号、标题、场景分隔、对白字体规则；
- 提供上一章、下一章、返回关系 Wiki；
- 章末附一个克制的“本章线路”卡片，但不抢正文注意力；
- 结局内容使用折叠提示或明确剧透标记。

章节完整编辑容器仍在幕后档案中可见，因此没有丢失“全部文档”的要求。

### 4.3 第二部分：幕后档案

建议路径：

```text
/handbook/archive/
```

按源目录分层展示：

```text
项目说明
├── README / TASKS / project
人物
素材与线索
剧情与支线
世界设定
章节编辑容器
章节大纲与揭示账本
实施与盲测记录
```

历史实施稿保留，但在标题附近显示“开发记录，不代表当前可玩数据”。默认关系图不纳入这些页面，避免 100 节点版和 40 节点版混成一个网络。

## 6. Quartz 技术方案

### 6.1 版本与目录

以实施时锁定的 Quartz 5 精确版本为基线，不使用浮动 `latest`。官方当前文档为 Quartz 5.0.0，要求 Node 22+ 和 npm 10.9.2+；本机 Node 25.6.1 / npm 11.9.0，远端 Node 24.13.1，均满足要求。

建议仓库结构：

```text
telephone/
├── story-outline/               # 唯一故事 Markdown 源
│   └── wiki/                    # 新增的人工作者整理层
├── handbook-site/              # Quartz 5 工程、主题、组件和锁文件
│   ├── content/                # 构建生成的暂存内容，不手改
│   └── public/                 # Quartz 输出，不手改
└── tools/handbook/             # 同步、校验与索引生成脚本
```

Quartz 工程使用独立 `package.json` / lockfile，避免把 Quartz 依赖混入当前 Vite 游戏 bundle。根目录只增加便捷脚本，例如：

```text
handbook:sync
handbook:check
handbook:dev
handbook:build
```

### 6.2 确定性内容同步

`handbook-site/content/` 是构建暂存层，不是第二套故事源。同步脚本完成：

1. 清理上一次生成的暂存内容；
2. 复制筛选后的74个作者源文档；
3. 排除模板、`AGENTS.md`、`style/`、`styles/`、`published/`、`handbook-site/internal/` 和非内容文件；
4. 给生成副本补充网站专用 frontmatter：`surface`、`canon_tier`、`spoiler`、`permalink`、`cssclasses`、`publish`；
5. 从六个章节抽取 `## Draft` 生成小说阅读页；
6. 从活动 JSON 生成号码、来电、物件和结局索引数据；
7. 验证人工 Wiki 对这些稳定 ID 的引用，发现漂移就终止构建。

同步过程不得反向改写原 Markdown。需要纠正故事事实时直接编辑 `story-outline/` 源文件，再重新生成。

### 6.3 Quartz 能力使用

第一版使用以下原生能力：

- Wikilinks 与 backlinks：连接人物、号码、物件、章节和结局；
- Graph：只在 Wiki 人物/物件/线路页显示局部关系图，首页可打开全局 Wiki 图；
- Mermaid：绘制确定性的拨号流程、章节流转和结局分叉；
- Explorer 与文件夹索引：分层展示 Wiki、故事与幕后档案；
- Search：同时搜索中文名、英文名、别名和号码；
- Reader Mode：正文和长篇幕后文档进入无侧栏阅读；
- Frontmatter / aliases / permalinks：稳定 URL 与旧名称跳转；
- Explicit Publish 与 ignore patterns：只发布同步脚本允许的内容，避免未来模板或内部文件意外进入网站；
- 本地字体与静态资源：避免依赖第三方字体服务。

Graph 的连接来自明确的 wikilink。它适合展示“谁与当前条目相连”，但不擅长解释每条边的语义，所以剧情顺序与结局条件仍用 Mermaid 和表格。只有在内置 Graph 无法清楚过滤人物/物件/号码时，才增加一个小型“关系交换台”组件；不先造一套新的图数据库。

### 6.4 自定义组件

第一版控制在以下五类组件内：

1. `HandbookCover`：员工手册封面、十九号副本与三类入口；
2. `CanonBadge`：只在幕后显示已部署 / 扩写 / 开发记录；
3. `EntityDocket`：人物、电话、物件的统一案卷卡；
4. `TelephoneDirectory`：从活动 JSON 索引生成的电话簿；
5. `ChapterNavigator`：正文上一章 / 下一章 / 本章关联线路。

自定义脚本遵守 Quartz SPA 的 `nav` / `render` 生命周期并清理事件监听。所有 CSS 使用 `telephone-handbook-*` 命名空间，因为 Quartz 组件样式是全局样式，不依赖 CSS Modules。

## 7. 视觉与交互方案

### 7.1 共用视觉语言

从当前游戏继承而不是另造品牌：

- 黑色胶木电话与深墨绿面板；
- 电话亭暗红边框；
- 氧化黄铜与旧金色；
- 奶油纸、褪色墨水、红色归档章；
- LCD绿只用于状态和号码，不作为大面积装饰；
- `Gill Sans / Avenir / 系统无衬线`、`Georgia / Songti SC`、`Courier New` 三组字体角色。

首版不依赖新的 AI 插画。优先复用现有 CSS 纹理、色板和已授权的项目资产；如果以后需要封面或插图，再另行提出视觉资产任务。

### 7.2 Wiki：拟物但可读

Wiki 首页像一本放在电话亭柜台上的夜班手册。人物页像员工卡与监听记录，电话页像电话簿索引，物件页像证物袋或维修回执，关系图像总机插孔。

拟物效果只用于层级和身份，不遮挡正文：

- 纸张倾斜最多 0.5–1 度；
- 雨线与灯光只做低对比背景；
- 图谱、搜索和正文保持标准点击区域；
- 不加入自动播放音频；
- `prefers-reduced-motion` 下关闭雨动效、入场位移和脉冲灯。

### 7.3 故事：书页质感

正文区使用克制的暖白纸、窄行宽和明显段落节奏：

- 正文宽度约 65–72 个中文字符；
- 叙述遵守项目的无衬线规则；
- `*“对白”*` 使用衬线斜体；
- 场景分隔沿用 `---` 的语义；
- Reader Mode 隐藏目录和图谱；
- 手机端不模拟双页书，保持单列阅读。

### 7.4 幕后：记事本质感

幕后档案采用灰米纸、细横线、页签、打字机元数据和少量红章。它应比 Wiki 更像编辑工作台，但避免把每一页做成不同道具。

## 8. 子路径与宝塔部署方案

目标 URL：

```text
https://05-telephone.seeds100.bowen.wang/handbook/
```

Quartz `baseUrl` 配置为：

```text
05-telephone.seeds100.bowen.wang/handbook
```

Quartz 输出是静态 HTML/CSS/JS。推荐让宝塔 Nginx 直接服务它，而不是让现有 Vite Preview 处理：

1. 构建输出复制到服务器独立静态根下的 `handbook/`；
2. Nginx 为 `/handbook/` 设置优先于现有应用反向代理的静态 `location`；
3. 使用 Quartz 官方建议的 `try_files $uri $uri.html $uri/` 规则支持无 `.html` 链接；
4. 其余 `/`、`/admin`、`/record` 等继续进入当前 Telephone 服务；
5. 对带内容哈希的 CSS/JS 使用长期缓存，对 HTML 使用短缓存；
6. 验证首页、深层页面刷新、图谱索引、搜索索引和所有静态资源在子路径下正常。

## 11. 验收标准

### 内容

- 首页第一次阅读不需要理解 Seedline 旧实施历史；
- 当前可拨电话簿只列 8 个活动号码；
- 当前场景只列 10 件活动物件；
- 人物对应存在证据等级，不把猜测写成事实；
- 关系 Wiki、小说正文和幕后原文之间没有手工复制导致的分叉；
- 过期实施资料仍可查，但始终带开发记录标记；
- 连续性检查为 0 error / 0 warning。

### Quartz 与交互

- `/handbook/` 和所有深层 URL 可直接刷新；
- Wiki 有人物关系、物件关系、电话簿、流程和结局入口；
- Graph 只显示有意义的 Wiki 关系，不被全部实施文档淹没；
- Mermaid 流程与当前活动号码一致；
- 正文 Reader Mode 清晰，手机端可连续阅读六章；
- 搜索中文、英文名、alias 和号码均有结果；
- 不加载未授权第三方追踪或外部字体。

### 回归

- 当前 Telephone 游戏测试、Lint 和生产构建继续通过；
- Quartz 依赖不进入游戏客户端 bundle；
- 宝塔原有游戏、后台和档案路由无回归；
- 静态站回滚不需要回滚活动游戏数据。
