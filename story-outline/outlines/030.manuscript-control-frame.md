# 六章正文重写控制框架

## Narrative Engine

- **主要引擎：** 从一项公共服务事故追到技术治理权的制度谜案。
- **次要引擎：** 合法但冲突的职业需要；限制明确的工作记忆分区与分布式线路技术。
- **排除：** 时间倒流、人格上传、全知机器、所有来电属于Vale、中央一键删除全部地方副本。

## Promise

读者会看见一项曾经有益的技术如何通过合同、绩效和版本控制变成制度伤害，并陪伴其创造者的夜班工作自我决定谁有权发布下一版。所有抽象问题都必须先落到病人、工资、执照、保管链或更新窗口。

## Change Chain

病人转运受阻 → 原工单进入公司 → 员工为保留证据承担纪律风险 → 正式故障打开人工审计 → Vale工作身份被证明 → 外部托管恢复安全签名来源 → 原声揭示创始责任和分布式风险 → 人物授权汇流 → 四点版本选择。

## Cost Ladder

1. 错误分类让一名病人的转运延误。
2. 保留工单让Peter承担纪律风险；删除工单让Maeve失去证据。
3. 公开身份使Vale接近整合，也使Wren取得更多声学样本。
4. 外部托管放行会把Vale安全签名重新带入发布争议。
5. 原声让Vale不能再把分区伤害完全归给公司。
6. 最终任何选择都会失去完整记忆、统一更新、员工安全、证据资格或普通生活中的至少一项。

## Prose Rules

- 每个场景开头用动作或工作问题建立“谁、哪里、现在要什么”，不以专有名词开场。
- 一次对话最多解释一个新术语；角色必须在同一场景用它改变决定。
- Wren的台词以指标、合同、总体风险和可执行交换为主；不发表抽象控制宣言。
- Maeve使用病人状态、路线和时间；Peter使用班次、福利、工单和处罚；Leonard使用现象、证据和签字；Dorothy使用来源、保管链和权限。
- 重大揭示后必须先展示行为变化，再进入下一揭示。
- 分支标题只用于真正不同的夜班或结局；正文不向读者说明“现在进入互斥状态”。

## Resolution Gate

第六章只可使用：医院事件记录、Peter员工令牌、Leonard技术签字、Dorothy外部托管、Vale安全签名、Wren运营签名、凌晨同步和已经展示的地方副本。任何新能力均为连续性错误。
